# LINE Bot + LIFF + GAS 実装タスク（TDD版）

## 方針
- 各タスクは `背景` → `失敗するテスト追加` → `実装` → `リファクタ` の順で進める。
- 完了条件は「仕様を満たすテストが通ること」を最優先する。
- テスト観点は必ず `正常系 / 異常系 / 境界値` を最低1ずつ含める。

## 共通テスト基盤（事前準備）
- 背景: GAS 実装が増える前提で、再現可能な検証ルートを先に固定する。
- 完了条件（TDD）:
  - [x] テストディレクトリと実行方法を定義する（`tests/` + 実行コマンド）。
  - [ ] 「空振りで失敗する」最低3件の受け入れテストを事前に定義する。
  - [x] CI/ローカル実行手順を `README.md` に追記する。

- 背景: 入口・デプロイ条件が安定しないと、以降の変更が再現不能になる。

  - [x] `appsscript.json` の権限（External Request/Spreadsheet）が不足時に失敗するテストを追加。
  - [x] 設定正常時に `clasp push` 実行前チェックと実行コマンドが失敗しないことを確認するテストを追加。
  - [x] 上記テストが全て通る状態でデプロイ準備完了。

## 2. データモデル（Spreadsheet）設計
- 背景: 「日次独立」「権限」「履歴」を支える唯一の保存構造。
- 完了条件（TDD）:
  - [x] 主要シート構成（stores/users/line_accounts/checklist_runs/...）の存在チェックテスト。

  - [x] データ不正時の初期化・ロールバックテスト。

## 3. LINE連携・認証基盤（`/api/link`）
- 背景: LINEユーザーID単体で権限判断しないため、内部ユーザー紐づけが必須。
- 完了条件（TDD）:
  - [x] LIFF認証コンテキストからLINEユーザーIDを取得できない場合に紐づけ失敗するテスト。
  - [x] 正常な `employeeCode + passcode` でリンク成功するテスト。
  - [x] 重複リンク時（既存ユーザー/別ユーザー）で衝突検知テスト。
  - [x] リクエストボディが `employeeCode + passcode` のみのとき受理され、`lineUserId` を含む場合は拒否するテスト。
  - [x] `GET /api/me` で role/store が返るテスト。

## 4. LINE Bot入口（Webhook + リッチメニュー誘導）
- 背景: 利用導線（メニュー→LIFF）が機能しなければ運用開始できない。
- 完了条件（TDD）:
  - [x] 有効署名を持つWebhook受信時に処理が進むテスト。
  - [x] 不正署名時に403/拒否されるテスト。
  - [x] メニュー選択イベントでLIFF誘導URLが返るテスト。
  - [x] 利用者/管理者いずれも共通4メニュー（今日のチェックリスト・未完了一覧・履歴を見る・ヘルプ）が返るテスト。

## 5. LIFF共通基盤実装
- 背景: 画面間で重複する初期化・権限取得・エラー処理を共通化する。

  - [x] 再読込/更新ボタンが状態取得APIを呼ぶテスト。

## 6. アルバイト向けチェック機能
- 背景: 本体価値の中核。日次共有状態の確認・チェック入力。

  - [x] チェック未実施項目が `check` 後に checked に遷移するテスト。

  - [x] 同時押下など競合ケースの冪等性テスト。

## 7. チェック取消と権限制御
- 背景: 不正取消防止と監査整合性のため。
- 完了条件（TDD）:

  - [x] 管理者が他人のチェックを取消できるテスト。
  - [x] 取消時に `action=uncheck` の履歴ログが1件増えるテスト。
  - [x] 取消後の状態がUIに反映されるテスト。

## 8. 管理者向けテンプレート/項目管理
- 背景: 運用要件として項目を柔軟に更新できる必要がある。
- 完了条件（TDD）:
  - [x] 管理者でないユーザーが項目追加時に拒否されるテスト。
  - [x] テンプレート新規作成・更新・項目追加・編集・削除のCRUDが成功するテスト。
  - [x] 無効値（空タイトル等）でバリデーションエラーになるテスト。
  - [x] 変更履歴（edit/delete）ログが残るテスト。

## 9. 日次ジョブ（10:30）
- 背景: 当日分を毎日0件で開始できることが運用成立条件。
- 完了条件（TDD）:
  - [x] 当日分が未作成時に作成されるテスト。
  - [x] 既存時は二重作成しないテスト。
  - [x] 全項目が unchecked で初期化されるテスト。
  - [x] 失敗した通知が `notifications` に failed で残るテスト。

## 10. 日次ジョブ（0:00締切）
- 背景: 前日分の締切と未完了通知が要件の中核。
- 完了条件（TDD）:

  - [x] 未完了なし時は通知抑止されるテスト。
  - [x] 締切後状態が `closed` に遷移するテスト。
  - [x] 締切後チェックが履歴で「締切後」と識別可能なテスト。

## 11. 通知失敗の監査
- 背景: 運用トラブル時の調査可能性を担保する。
- 完了条件（TDD）:

  - [x] 通知再試行時に同一対象重複送信しないテスト。
  - [x] 管理者の手動通知API（`POST /api/admin/checklists/{runId}/notify-incomplete`）が管理者のみ実行可能なテスト。
  - [x] 手動通知成功時に `notifications.type=manual_reminder` が保存されるテスト。
